二、第一部分：Context Engineering技术详解
（1）核心概念：为什么需要Context Engineering
语言模型的本质是做文字接龙：收到输入prompt后，基于前文内容输出下一段文字。当AI Agent需要调用外部工具时，工具产生的输出，必须和之前人类给出的原始指令、语言模型操控工具的指令拼接在一起，才能重新送给语言模型继续处理。

但语言模型的输入长度（上下文窗口）是有限制的，AI Agent作为语言模型和工具交互的中间层，必须主动选择合适的内容送给语言模型、控制输入总长度，这个管理过程就是Context Engineering（上下文工程）。

（2）两种处理模式对比

3. 直接把Iₜ和Oₜ追加到context后，更新为Cₜ₊₁进入下一轮循环	整体流程和无处理一致，唯一改变的是最后更新上下文的步骤：
不再直接追加内容，而是使用复杂处理函数F对当前上下文做处理，F最核心的功能就是对过长的历史记录做压缩，把长历史压缩为更短的内容，保证总长度不超出上下文窗口限制

（3）常见压缩实现方式
LLM summary（大模型摘要压缩）：直接让语言模型把历史记录中扣除system prompt的较早内容，重新输出一份摘要，缩短总长度。但在多个实际测试中，它的效果和更简单的observation masking方法差别不大，没有显著优势，同时还会存在「轨迹延长」的问题（压缩后上下文长度依然会缓慢增长）。

observation masking（观测掩码）：直接把工具产生的输出，替换为一句占位文本（比如“这里曾经有一个工具输出”），不需要做复杂的摘要生成，操作极其简单。实际表现和大模型摘要方法接近，成本更低。

混合策略：在轨迹延长的前期使用observation masking，缩短工具输出占用的空间；当长度还是持续增长到阈值后，再使用大模型摘要做整体压缩；同时可以把压缩后的内容存储到外部硬盘中，需要的时候再读取调用，进一步控制当前prompt的长度。

（4）上下文分层管理方案
为了更高效地管理上下文，可以把整个上下文拆分为两类，分层存储：

P类：当前会直接送入语言模型处理的信息，已经加载在prompt中

M类：当前不需要处理，存储在外部硬盘中的信息，需要时再动态加载

每次更新上下文时，会分别更新当前P类信息Pt和M类信息Mt：需要调用存储在M类的内容时，读取到P类中；需要压缩当前P类时，把不常用的内容转存到M类中，实现动态分层管理。

（5）压缩带来的核心问题：Context Collapse（上下文坍塌）
压缩历史记录时，很可能会把关键信息误删除，导致语言模型因为缺失重要信息，把本来可以答对的任务做错。比如Meta的相关研究中，AI处理邮件任务时，压缩操作错误删除了用户给出的“只回复来自特定发件人邮件”的指令，最终导致AI输出完全错误。

针对这个问题，最新的ACON论文提出了优化方案：额外增加一个语言模型，为做摘要的语言模型生成反馈，帮助摘要模型更好识别哪些信息不能丢，提升摘要的准确性。在AppWorld基准测试中，使用ACON方法不仅能减少25%以上的token消耗，还能将任务正确率提高8个百分点左右。

（6）压缩时机的不同方案
阈值触发压缩：最直观的思路是当上下文长度超过预设阈值后，再触发压缩操作。但实际运行中，语言模型本身对“被抹除记忆”适配性不好，效果并不稳定。

固定规则触发：OpenClaw项目采用写死的规则，只要长度超过上限就触发压缩，工程实现简单，可落地性强。

模型自主学习压缩：AgentFold论文通过微调语言模型的参数，直接训练模型学会自主使用压缩记忆工具，让模型自己判断什么时候该压缩，适配性更好。

子智能体自主压缩：使用sub-agent（子智能体）本质就是一种自主压缩行为：主智能体把任务交给子智能体执行，子智能体执行完任务只返回最终结果，会直接抹除执行过程中的所有上下文，相当于主动完成了一次压缩。这种自主压缩能力需要通过训练获得，训练时会使用强化学习方法，同时加入两个惩罚项：主上下文过长惩罚、子智能体输出超出范围惩罚。

（7）从根源解决长度问题：输入过滤思路
统计发现，工具输出（观测内容）会占据上下文大约70%-80%的内容，因此最根本的优化思路，是在观测内容进入语言模型之前，就先做过滤，只保留真正有用的部分：

优化基础工具：比如改进read文件工具，让它只把文件中和当前任务相关的部分提取出来送入模型，不需要把整个大文件都放进上下文

按需动态加载：OpenClaw通过memory search和memory get两个工具实现对记忆的过滤，只在需要的时候加载对应的记忆内容，本质和检索增强生成（RAG）的思路一致

动态选择工具：MCP-Zero更进一步，提出让语言模型自己动态决定当前需要哪些工具，不需要把所有工具的描述都放在prompt中，进一步节省上下文空间

（8）前沿方向：Agentic Context Engineering
最新的研究方向是把上下文工程的决策权直接交给语言模型本身，让模型自主管理上下文：

Dynamic Cheatsheet：早期方案，直接用prompt engineering的思路实现上下文工程，不需要改动模型参数

Agentic Context Engineering论文方案：让语言模型自主维护一份playbook（操作手册），playbook的演化需要经过多个语言模型检查步骤，模型自己生成修改指令来更新playbook内容，实现动态进化

Recursive Language Model：把大部分上下文存在外部硬盘中，只把简短的元数据加载到当前prompt中，让语言模型自主通过RAG检索需要的内容，更新prompt。这个方案在多个长上下文基准测试中都取得了不错的效果。

三、第二部分：AI Agent之间的互动
当多个AI Agent同时存在时，会产生不同形式的互动，常见互动模式分为两类：

协作模式：多个Agent分工完成同一个复杂任务，比如一个Agent负责收集信息，另一个Agent负责整理结论，第三个Agent负责输出最终方案，通过分工提升复杂任务的处理效率。

竞争模式：多个Agent针对同一个问题分别给出方案，互相评审对方方案的问题，通过竞争辩论发现方案中的漏洞，最终得到更完善的结果。

工程实现上，多Agent互动本质还是在单Agent框架基础上做扩展，每个Agent都维护自己的上下文，互动过程就是Agent之间互相传递输入输出，每个Agent收到其他Agent的输出后，更新自己的上下文再继续处理，底层依然需要Context Engineering来控制每个Agent的上下文长度。

四、第三部分：AI Agent对未来工作的冲击
任务拆解重构：未来大部分复杂工作都会被拆解为多个子任务，每个子任务可以由AI Agent完成，人类只需要负责整合结果和做最终决策，工作流程会被彻底重构。

岗位需求变化：基础的执行类岗位需求会减少，而能够设计AI Agent工作流程、训练调教AI Agent的新岗位需求会快速增长。

能力要求改变：人类工作者的核心能力要求，会从“执行具体任务”转变为“给AI Agent设计目标、验证结果”，对人的抽象思考能力和判断能力要求更高。
